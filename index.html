The Multi User Domain project is an early-development open-source platform for creating Multi User Domains. Building on semantic web standards users store their own data, taking their characters and items with them as they travel between different "worlds"; servers created by a community to store shared physical world, like towns and places.

All resources (Characters, Buildings, Memories) are stored in RDF Graphs, a form of graph storage which uses URIs to uniquely identify all objects, classes (types) and properties. For example my character may be stored at `http://example.com/characters/alice/`, the RDF definition of what a character is being found at `http://example.com/vocabulary/#character` By describing our world in this way we can share definitions between worlds and start building this "infinite" expanse of interoperable worlds for characters to travel between

We separate the concerns between the storage of objects in our worlds (in RDF) and the description of them, by writing generic Content Server endpoints which can receive requests like "describe this object for me". My Content Server can analyse the class, object and properties it can find for things which it understands, and return text content to the user describing it. As a very simple example I can analyse a Stadium and reply "it's quiet" or "it's noisy" dependening on whether there is an event on. By returning _this_ in RDF, too, we can separate input stimulus like `mudcontent:sees` and `mudcontent:hears`, and decorate it with `mudcontent:because`. The client can then understand to an extent the content it is being given, and reason what to display based on the _quality_ of it (specificity for the user) and its relevance. It can even pipe the content to other servers to build on it and find something better

RDF data on the semantic web is open by design, making it highly extendible. There is a lot of fun to be had in the building of a world, even without any "text-based adventure", simply by creating a village, coming back in 6 months to find that your village is now a city, or that the bartender at the saloon you visited last week is complaining about another player who came in and challenged them to a duel. Your world server doesn't have to define what a duel is, what a saloon is or what a Memory is, it just needs to allow players and machines to write to it in RDF. It doesn't have to redefine how to describe what's happening because this concern has been separated to the content server providers.

The project is in incredibly early development, we basically have the very limited content description example "there is an event on at the stadium" used before. It's open-source and the quality of the game will depend entirely on the health of a community, so if you're interested to get involved in any capaciy then please get in touch.

The components of the platform are as follows:

The Vocabularies

The vocularies are OWL/RDF documents served in Turtle format to describe all of the data models being used to build the world, content and logic in the MUD. All in draft status and so subject to a high degree of change for the time being

Accessible at https://github.com/Multi-User-Domain/vocab

React - The Client

Currently this repository defines all of the React extensions for helping to interface with the MUD vocabularies and for building a client-side app to interface with MUD servers. It also implements a specific client for interfacing with them, which when the repository is more mature will be separated from this repository (https://github.com/Multi-User-Domain/mud-react/issues/2).

Accessible at https://github.com/Multi-User-Domain/mud-react

Java & Apache Jena - The Server(s)

The repository for all things World Server, Content Server and Action Server.

Accessible at https://github.com/Multi-User-Domain/mud-jena
